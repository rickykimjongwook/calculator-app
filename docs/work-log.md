## [LOG-1] 2026-04-02 — 프로젝트 기획 요청

### 프롬프트

```
클로드 코드를 이용해서 실제 애플리케이션을 빌드하고 싶어.
나는 프로덕트 매니저이긴 한데, 실제로 애플리케이션을 빌드하거나
웹 환경에 서빙해본 적이 없어.
목표는 이거야: '단순한 계산기 웹 앱'을 인터넷에 올린다.
이 과정을 위한 가이드를 제작해줘.
그러면 나는 그걸 보고 따라가며, 웹 애플리케이션을 만들어볼거야.
로컬 프로젝트 세팅, 깃허브 연동, 무료 호스팅 사이트 이용,
db 연동 등을 하고 싶어. 계정시스템은 일단 배제할 거야.
```

### 왜 이렇게 프롬프팅했나

- **계산기를 선택한 이유:** 너무 복잡한 앱은 초반에 막히면 포기할 것 같아서. 기능 명세가 완전히 명확한 것을 골랐다. 계산기는 누가 봐도 뭘 만들어야 하는지 알 수 있다.
- **계정 시스템을 뺀 이유:** 인증/보안 로직이 들어가면 복잡도가 급격히 올라간다. 핵심 흐름(빌드 → 배포 → DB)을 먼저 경험하고 싶었다.
- **"코드 짜줘"가 아니라 "가이드 제작해줘"라고 한 이유:** 클로드가 바로 코드를 짜버리면 내가 맥락 없이 따라가게 된다. 먼저 전체 그림을 이해하고 싶었다.
- **내 배경을 명시한 이유:** "PM이고 개발 경험 없다"는 정보를 주면 클로드가 설명 수준을 맞춰줄 것이라 기대했다.

### Claude Code의 반응

- **Plan Mode는 내가 처음부터 선택했다.** 코드가 엉뚱한 방향으로 진행되는 걸 막고 싶어서 의도적으로 Plan Mode로 세션을 시작했다.
- 기술 스택을 직접 제안: **Next.js + Vercel + Supabase**. 이유도 각각 설명해줬다 (Vercel-Next.js 궁합, Supabase 무료 tier 등).
- 6단계 구조의 가이드 계획안을 작성하고 "이 계획대로 진행할까요?" 형태로 승인을 구했다.
- 내가 승인하자 실행 모드로 전환.

### 배운 것 / 놀란 것

- **Plan Mode의 존재를 몰랐다.** Claude Code가 바로 실행하지 않고 계획을 먼저 공유하고 승인을 구하는 단계가 있다는 것. 이게 꽤 중요한 것 같다 — 엉뚱한 방향으로 작업이 진행되는 걸 막아준다.
- 내가 기술 스택을 전혀 몰라도, 맥락만 주면 적절한 스택을 추천해준다. "무료", "초보자", "빠른 배포"라는 암묵적 요구사항을 읽어냈다.
- **"나는 PM이다"라는 한 줄이 설명 방식을 바꿨다.** 기술적 설명에 이유(why)를 항상 붙여줬다.

### 다음에 쓸 수 있는 프롬프트 패턴

```
[나는 누구 + 경험 수준] + [최종 목표] + [원하는 산출물] + [제약 조건]
```

예: "나는 PM이고 개발 경험 없어. 계산기 웹앱을 인터넷에 올리고 싶어. 가이드를 만들어줘. 계정 시스템은 제외."

## [LOG-3] 2026-04-02 — 가이드와 로그 문서 분리

### 프롬프트

```
guide 문서에서 'log' 성격은 떼어내서, 별도의 문서로 만들자.
work log.md라고 하자. 나중에 콘텐츠를 만들기 위한 로그로서
충실하게 만들려고 해. 가이드는 그냥 '내가 따라할 가이드'로서 남기고,
워크로그는 '내가 나중에 콘텐츠 만들 재료'로서 충실하게.
```

### 왜 이렇게 프롬프팅했나

- 하나의 문서에 가이드와 로그가 섞이면 두 가지 목적 모두 어설퍼진다.
- **가이드:** 실습할 때 빠르게 참조할 수 있어야 한다 → 간결하고 명확해야 함
- **워크로그:** 콘텐츠 원재료가 되어야 한다 → 날것의 생각, 놀란 점, 실패한 것까지 상세히
- 두 문서의 독자가 다르다: 가이드는 '미래의 나(또는 따라하는 다른 PM)', 워크로그는 '콘텐츠를 만들 때의 나'

### Claude Code의 반응

- `guide.md`: 로그 섹션 제거, 단계별 가이드만 남겨 군더더기 없이 정리
- `work-log.md` (현재 이 문서): 새로 생성. 콘텐츠 재료로 충실하게 쓸 수 있도록 로그 형식 설계

### 배운 것 / 놀란 것

- **문서의 목적을 명확히 하는 것이 중요하다.** "가이드"와 "로그"는 쓰임이 다르고, 독자가 다르고, 형식이 달라야 한다. 처음부터 이걸 나눠서 요청했으면 더 좋았을 것.
- Claude는 "충실하게"라는 추상적인 표현도 해석해서 구조를 제안한다. 단, 내가 원하는 방향을 잘 알려줄수록 결과물이 더 정확해진다.
- 문서를 분리하는 것은 단순한 정리가 아니라 **사고의 분리**다. 실행 중에는 가이드를 보고, 완료 후에는 로그를 채운다.

### 다음에 쓸 수 있는 프롬프트 패턴

```

예: "현재 문서에 A와 B가 섞여 있어. A는 [목적1]을 위한 거고, B는 [목적2]를 위한 거야. 두 파일로 나눠줘."

## [LOG-6] 2026-04-02 — 훅 범위를 글로벌 → 프로젝트 한정으로 변경

### 프롬프트

```
이 훅을 나는 '계산기 웹 애플리케이션 만들기' 지금 프로젝트를 하는 동안에만
실행시킬거야. 미래에 계속 쓸 의도는 없어. 그렇다면 어떻게 해야 할까?
```

```
바꿔줘
```

### 왜 이렇게 프롬프팅했나

- 훅 개념을 이해하고 나니 "이게 지금 내 모든 Claude 대화에 다 붙는 건가?"라는 생각이 들었다.
- 이 프로젝트가 끝나면 로그를 안 남겨도 되는데, 글로벌 설정이면 나중에 끄는 걸 잊을 것 같았다.
- 먼저 방법을 물어보고("어떻게 해야 할까?"), 설명을 들은 후 실행을 요청("바꿔줘")했다. 2단계 프롬프트.

### Claude Code의 반응

**1단계 — 방법 설명:**
- 설정 파일에는 범위가 다른 3가지가 있다고 설명:
  - `~/.claude/settings.json` → 글로벌 (모든 프로젝트)
  - `.claude/settings.json` → 프로젝트 전용
- 훅을 글로벌에서 프로젝트 파일로 옮기면 해당 폴더에서만 실행된다고 안내

**2단계 — 실행 ("바꿔줘"):**
- Plan Mode 진입 → 변경 계획 수립 → 승인 후 실행
- `~/.claude/settings.json`에서 훅 블록 제거
- `calculator-app-guide/.claude/settings.json` 새로 생성하여 훅 이동

### 배운 것 / 놀란 것

- **설정 파일에는 범위(scope)가 있다.** 글로벌/프로젝트/로컬 세 단계. 좁은 범위가 넓은 범위를 덮어쓴다.
- "어떻게 해야 할까?" → "바꿔줘" 패턴이 효과적이다. 먼저 방법을 이해하고 승인한 뒤 실행시키면 예상치 못한 결과를 방지할 수 있다.
- 프로젝트가 끝나고 폴더를 지우면 훅도 함께 사라진다. 별도로 정리할 필요 없다.
- **Plan Mode**가 또 등장했다. "바꿔줘" 한 마디에도 바로 실행하지 않고 계획을 먼저 보여주고 승인을 구했다. 설정 파일 변경처럼 되돌리기 까다로운 작업일수록 이 패턴이 안전하다.

### 다음에 쓸 수 있는 프롬프트 패턴

```
[상황 설명 + 의도] + "어떻게 해야 할까?" → 설명 확인 → "바꿔줘" / "해줘"
```

## [LOG-10] 2026-04-02 — GitHub Repository 생성 및 push (PAT 토큰 인증)

### 프롬프트 (행동)

터미널에서 직접 입력:
```

push 시도 → username/password 요청 → 비밀번호 입력 실패 → Claude에게 질문:
```

그런데 나는 password말고 google oauth로 깃허브를 이용하고 있어.
oauth 인증을 터미널에서 대신할 수는 없나?
```

### 왜 이런 상황이 생겼나

- GitHub을 Google OAuth로 가입했기 때문에 별도의 비밀번호가 없다.
- Git은 터미널에서 HTTPS 방식으로 인증할 때 비밀번호를 요구하는데, OAuth 계정은 이 방식이 안 된다.
- 게다가 GitHub은 2021년부터 비밀번호 인증 자체를 막아두고 있다. 비밀번호가 있어도 사용 불가.

### Claude Code의 반응

- **Personal Access Token(PAT)** 을 사용하라고 안내.
- GitHub Settings → Developer settings → Tokens (classic) 경로로 토큰 발급 방법 설명.
- push 시 Password 칸에 토큰을 붙여넣으면 인증된다고 안내.

### 결과

- PAT 발급 후 push 성공.
- GitHub에서 `rickykimjongwook/calculator-app` repository에 파일 확인됨.

### 배운 것 / 놀란 것

- **GitHub 터미널 인증 = PAT.** 비밀번호가 아니다. OAuth 계정이든 일반 계정이든 터미널에서는 PAT를 써야 한다.
- **PAT는 비밀번호처럼 쓰이지만 성격이 다르다.** 특정 권한(repo 읽기/쓰기 등)만 부여할 수 있고, 만료일을 설정할 수 있고, 언제든 폐기할 수 있다. 실제 비밀번호보다 더 세밀하게 관리 가능.
- **PAT를 프로젝트 폴더 안에 저장하면 실수로 GitHub에 올라갈 수 있다.** Claude가 이 위험을 먼저 경고하고 `.gitignore`에 추가할 것을 안내했다.
- 토큰을 어디에 저장할지 헷갈렸다 — 프로젝트 폴더, 별도 폴더 등 여러 시도를 거쳐 결국 `calculator-app-guide` 폴더(GitHub 연동 안 된 곳)에 보관하는 걸로 결론.

- 이번에는 두 가지를 순서대로 모두 진행: Redeploy(환경변수 반영) → git push(코드 반영)

- 비로그인: `localStorage.getItem/setItem("calc_history")` — 브라우저 캐시에 저장, 최대 20개 보관
- 로그인: Supabase DB에 `user_id` 포함하여 저장. `onAuthStateChange` 리스너로 상태 변환 시 자동 전환

**Auth 세션 유지:**
- `supabase.auth.onAuthStateChange` — 로그인/로그아웃/토큰 갱신 이벤트 구독
- `supabase.auth.getUser()` — 마운트 시 기존 세션 복원
- 새로고침해도 로그인 상태 유지됨

### 수동으로 해야 할 Supabase 설정

코드만으로는 할 수 없는 작업 2가지:

1. **Supabase Dashboard → Authentication → Providers → Email → "Confirm email" OFF**
   - 이 설정 없이는 가입 후 인증 이메일 발송 → 즉시 로그인 불가
   
2. **SQL Editor에서 실행:**
```sql
-- user_id 컬럼 추가
ALTER TABLE calculations ADD COLUMN user_id UUID REFERENCES auth.users(id);

-- 기존 anon 전체 허용 정책 제거
DROP POLICY IF EXISTS "Allow anon select" ON calculations;
DROP POLICY IF EXISTS "Allow anon insert" ON calculations;

-- 로그인 사용자 본인 기록만 허용
CREATE POLICY "Users can select own calculations"
ON calculations FOR SELECT USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own calculations"
ON calculations FOR INSERT WITH CHECK (auth.uid() = user_id);
```

### 배운 것 / 놀란 것

- **`onAuthStateChange` 콜백에서 클로저 함정**: `setUser(currentUser)` 직후 `fetchHistory(user)`를 호출하면 React 클로저로 인해 이전 `user` 값을 참조한다. 반드시 `fetchHistory(currentUser)`처럼 파라미터로 직접 전달해야 한다.
- **Supabase Auth는 이메일이 필수**지만, 이메일을 `{id}@calc.app`처럼 패턴화하면 "ID만 입력" UX를 만들 수 있다. 실무에서도 이런 패턴이 종종 쓰인다.
- **Plan Mode의 가치**: 복잡한 기능 추가 전에 계획을 먼저 검토하니 구현 중 방향 수정이 없었다. 특히 "Supabase 이메일 인증 비활성화 필요"처럼 수동 작업이 필요한 부분을 사전에 파악하는 게 핵심.

### 상황

LOG-14 코드 작업 완료 후 Vercel 배포에서 오류가 3번 연속 발생.

### 오류 1: 상단 바가 아예 안 보임

**원인:** 이전 커밋(`feat: supabase DB 연동`)에서 루트 레벨 `app/page.tsx`가 git에 올라가 있었음. 새로 만든 `src/app/page.tsx`와 두 파일이 공존하면서 Vercel이 구버전을 사용.

**해결:** `git rm app/page.tsx app/layout.tsx app/globals.css app/favicon.ico lib/supabase.ts`로 루트 레벨 구버전 파일 삭제 후 push.

---

### 오류 2: `Module not found: Can't resolve '@/lib/supabase'`

**원인:** `src/lib/supabase.ts`가 로컬에는 있었지만 git에 커밋이 안 된 상태. Vercel은 git 파일만 빌드하므로 파일 자체를 찾지 못함.

**해결:** `git add src/lib/supabase.ts` 후 커밋.

---

### 오류 3: 같은 오류 반복 — `Module not found: Can't resolve '@/lib/supabase'`

**원인:** `src/lib/supabase.ts`는 추가됐지만 `src/app/layout.tsx`, `src/app/globals.css`, `src/app/favicon.ico`도 누락되어 있었음. 이 파일들 추가 후에도 `@/` alias가 Vercel 빌드 환경에서 정상 해석되지 않는 문제가 남아 있었음.

**해결:** import 경로를 `@/lib/supabase` → `../lib/supabase` (상대경로)로 변경.

### 근본 원인 정리

프로젝트 초기에 `src/` 없이 루트에 `app/`, `lib/`을 만들었다가 나중에 `src/` 구조로 이전하면서 발생한 파일 관리 문제. **git에 추적되지 않은 파일은 Vercel에서 없는 것과 같다.**

### 배운 것

- **`git ls-files`로 배포 전 추적 파일 목록을 확인하는 습관**이 필요하다. 로컬에서 `npm run build`가 성공해도, git에 없는 파일은 Vercel에서 없는 것과 같다.
- **`@/` alias는 로컬에서는 동작해도 Vercel에서 실패할 수 있다.** tsconfig `paths` 설정이 있어도 빌드 환경에 따라 해석이 다를 수 있으므로, 문제가 생기면 상대경로로 바꾸는 것이 확실한 해결책.
- **구조 이전 시 구버전 파일을 git에서 명시적으로 삭제해야 한다.** 로컬에서 파일을 옮기거나 삭제해도 `git rm`을 하지 않으면 git은 여전히 구버전을 추적한다.
